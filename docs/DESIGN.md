# audio-debug Skill 设计文档

**作者**: kainan | **更新**: 2026-07-01 | **Skill 版本**: 1.2.0

## 设计目标

日志驱动的 AAOS 音频故障定位——从用户给出的日志出发，定位根因、给排查建议，必要时 adb 活体调试补充现场信息。主线是 log→根因 的循环；资产（paths.md / reference）在功能开发时也可直接查用。

## 核心设计决策

### 1. 平台先行

SOC 平台必须由用户给出（不从日志推断）。日志主体标签与所选平台矛盾时先向用户确认——平台错则标签表与源码路径整套错。

### 2. 源码可达性定模式

- **可达模式**：Read 到 paths.md 登记的源码文件 → 给文件路径 + 实际行号。
- **不可达模式**：正式模式而非兜底——用证据链 + 基于 paths.md 推断的可疑路径定位，精度到路径不到行，标「源码未验证」。
- **漂移 fallback**：paths.md 过时时用 Glob 在源码树按文件名定位，找得到仍算可达。

### 3. 路径相对 Skill 目录

所有相对路径相对 `${CLAUDE_SKILL_DIR}` 解析，与会话 cwd 无关——任意目录启动 `claude` 均可工作。

### 4. 文档分流（闸门）

每个独立问题一个文档，按根因确定度分类型：
- `fix_plan`（根因确定 + 源码可达）→ 步骤 4 全量审查 → 可进 5/6/7
- `analysis`（根因待证或源码不可达）→ 步骤 4 轻量审查 → 止于此交付

### 5. 改代码须用户同意

步骤 5 起用 AskUserQuestion 征得明确同意才动真实源码；真实工程改动不可逆，逐条确认、只改方案内列明项。

### 6. 审查循环到审干

审查是循环不是单遍——三维度（完整性/合理性/正确性）循环到「连续一轮零新问题」才停；安全上限 5 轮，未决项交用户裁决。
