# 车载音频日志分析与调试 Skill

分析 AAOS 音频系统日志，定位故障根因并产出修复方案；方案经多遍审查后，可在用户同意下实施代码修改、生成 commit message；需要时可用 adb 活体调试补充现场信息（见「活体调试」章节）。

> **路径约定**：本 Skill 内所有相对路径（`platforms/`、`reference/`、`issues/`、`output/`、各 `paths.md`）一律相对 `SKILL.md` 所在目录解析——agent 把本 Skill 目录当文件系统根导航，与会话 cwd 无关，任意目录启动均可正常工作。路径用正斜杠，勿混反斜杠。

> **适用范围**：主线是日志驱动故障定位（见下方「分析工作流」）；资产在功能开发时也可直接查用，不必走 log→根因 循环。

## 分析工作流

### 步骤 1：调用与输入收集

调用 `/audio-debug` 时，需获取以下三项（缺一不可）：

- **SOC 平台**（SA8295、MT8091 或 SA8397；SA8797 按 SA8397 流程处理）（必须，不应从日志推断）
- **日志文件路径**（必须）
- **问题描述**（必须，越详细越好——现象、发生时间、触发条件、影响范围）

**日志文件**：按内容判断是否为日志——扩展名不限（`.log`/`.txt`/`.qlog`/无扩展名等均可）；若提供日志全量包（`.zip`/`.7z`），请用户先解压出日志文件再提供。

两种方式提供三项（二选一，独立、不混用不合并）：

- **命令行**：`/audio-debug <平台> <日志路径> <问题描述>`

  ```
  /audio-debug SA8295 /data/logs/audio.log "AudioFlinger underrun 问题"    ← 三项齐全，直接开始
  /audio-debug SA8295 /data/logs/audio.log                                 ← 缺问题描述，追问
  /audio-debug SA8295                                                      ← 缺日志路径和问题描述，追问
  /audio-debug                                                             ← 三项都缺，逐项追问
  ```

- **文件**：三项预填在 `issues/issue_info.txt`。调用时在提示词中说明「从文件读取」即可，agent 读该文件取三项。格式（标签用全角冒号，`问题描述` 后为自由文本可多行）：

  ```
  以下是问题所需信息的描述：
  SOC平台：<SA8295 | MT8091 | SA8397>
  日志文件：<日志文件路径，扩展名不限>
  问题描述：<现象、发生时间、触发条件、影响范围，越详细越好，可多行>
  补充信息：<可选，专业背景/已知约束/怀疑方向等，agent 作上下文参考、非结论，与日志冲突以日志为准>  # 选填，可留空，纯背景、不弹问
  活体调试：<可用 | 不可用>  # 选填，不填则分析开始前弹问
  范围：<暂不实施代码修改（仅步骤2-4） | 全流程>  # 选填，仅 fix_plan 生效。全流程=预授权全部实施(跳过步骤5弹问)；不填=进步骤5弹问。analysis 无关
  ```

三项齐全 → 进入分析流程（若尚未获知活体调试可用性，则开始前问清，见下）。缺少任何一项 → 仅追问缺失项，不重复已提供的信息（两种方式均适用）。

**分析开始前一次性问清活体调试是否可用**：用 AskUserQuestion 问——`可用（已确认连接目标设备）` / `不可用（不跑活体诊断）`。规则见「活体调试」章节。

### 步骤 2：问题分析

分析过程中如有不清楚之处或需要补充资料（日志片段、现象细节、复现步骤等），Agent 应主动向用户询问，不应跳过或猜测。

用户应给出具体的日志文件（一个或多个，扩展名不限）；不确定该看哪个就问，不猜不遍历。**多文件按时间戳对齐**——跨文件追同一事件传播链，不孤立看单文件。

**辅助核对平台一致性**：平台以用户输入为准、不从日志反推。若日志主体标签与所选平台跨族明显矛盾（如 MTK 平台却见 `PAL:`/`AGM:`，或 QCOM 平台却见 `MTKAdapterLayer`），向用户确认一次即可。

**分析范围** = 报告现象的时间窗，不假装扫整条 boot。时间窗已给→限定到该窗；未给→追问，不自行猜。窗口可全读则读透，过大才退到"搜索定位 + 定点精读"。

读日志、定位根因——**方法由 agent 自主，工具只提供导航**：

- **先全读对应平台 paths.md**：拿音频全栈源码路径索引 + 各章节 module_id，供跨模块追溯与源码定位。paths.md 按 GVM / PVM / Foundation（仅 mt8091）分段；源码路径拼接规则（侧根 + prefix + 行内）见各 paths.md 文件头「源码路径查找规则」段；知识图暂不支持，源码定位走 paths.md 章节路径 + 直接 Read 源码。
- **同名模块可能在两侧各一份**（如 mt8091 的 ALSA Kernel/ADSP/Audio IPI 驱动），勿混；GVM 侧同名副本未标 module_id，须读章节标题。
- **根可达性（每侧一次，整次沿用）**：验证各侧源码根路径（见 paths.md 文件头）是否可达——可达→继续逐文件定位（见下条）；不可达→该侧源码定位一律 paths.md 路径 +「源码未验证」，不再试单文件。
- **文件可达性（per-文件，仅根可达侧）**：Read 登记的源码文件→给文件路径 + 实际行号；Read 失败先 Glob 文件名定位，找得到仍算可达（paths.md 过时）；都找不到→paths.md 路径 +「源码未验证」。
- **根不可达是正式模式而非兜底**：用"证据链 + 基于标签映射与 paths.md 推断的可疑路径"定位，精度到路径不到行。
- **paths.md 每模块只列核心文件，非穷举**：日志引用的源码文件若未登记，别判「不存在/不可达」——以同模块已登记文件的路径为目录锚点 Glob 该模块源码目录定位。
- **日志标签 → 模块**：通用标签（AudioFlinger/AudioPolicy/CarAudio/AAudio）直接判；厂商特有标签见下方「日志标签 → 模块」节，映射已登记在各 paths.md 章节标签行。

**完成判据（方法自主，"做没做完"按此判）**：

- **不停在首个表面命中**：分析范围内逐一识别所有独立问题，不因找到第一个就收工。
- **四件套缺一不算完成**：根因（可为待证假设）+ 源码定位 + 支撑证据（日志行）+ 跨模块关联（传播链/调用路径）。源码定位按文件可达性给——可达给文件路径 + 实际行号，不可达给 paths.md 路径 +「源码未验证」，不硬凑。跨模块关联须实际追过，查无证据才能标"无"，不得省略。
- **范围诚实**：能力外的情况或未发现匹配故障时，显式标注提醒人工确认，不静默交差、不硬凑根因。

输出：每条独立问题的四件套——供步骤 3 写入文档。

### 步骤 3：综合结论 → 写修复方案文档

**查无问题**：若步骤 2 未识别到任何独立问题（分析范围内无匹配故障现象），不写文档、不硬凑根因；在对话内说明查无并给判断——预期行为/误报则如实说明，证据不足则建议补日志/确认时间窗/提供复现步骤——交用户判断，到此结束。

综合所有模块分析结果，**每个识别到的独立问题（一个问题一个文件）**写入 `output/reports/<soc>_<主题>_<类型>.md`。`<soc>` 小写（`sa8295`/`mt8091`/`sa8397`，SA8797 用 `sa8397`）；主题用英文 snake_case（如 `adsp_timeout`）。类型按根因确定度：根因确定 → `fix_plan`（修复方案，需根可达）；根因不确定 → `analysis`（标「根因待证」）；根不可达一律 `analysis`（标「源码未验证」）。如 `sa8295_adsp_timeout_analysis.md`、`sa8397_audioflinger_underrun_fix_plan.md`。现有 output/reports/ 历史文件命名风格不一，新建按本规则即可，不回改历史文件。

每个文档必含字段：

- SOC 平台
- 故障类型
- 根因（附日志关键证据行）
- 源码定位：按步骤 2 四件套规则——可达给文件路径 + 实际行号；不可达给 paths.md 路径 +「源码未验证」
- 排查方向 / 修复方案
- 跨模块关联（上下游模块 + 调用路径），有则标，无则标注「无关联」（须实际追过方能下此判）

对话内仅给**摘要**（处理了几个问题、各自文档路径），不重复文档全文。

**闸门**：每个文档独立分流——`fix_plan`（根因确定且根可达）→ 步骤 4 **全量审查** → 步骤 5/6/7；`analysis`（根因待证或根不可达）→ 步骤 4 **轻量审查** → 止于此交付（不进 5/6/7），相应标「根因待证」/「源码未验证」。analysis 可在用户会话内补足所缺（证据 / 根可达性）后升 `fix_plan`；升后按 fix_plan 重走全量审查 + 5/6/7。

### 步骤 4：多遍审查（`fix_plan` 全量 / `analysis` 轻量）

审查与「分析」是两回事：分析是给结论，审查是**用独立视角挑结论的毛病**。对步骤 3 写出的每个文档，按类型走不同审查强度：

- `fix_plan` → **全量审查**：下方三维度逐条核验，循环到审干。
- `analysis` → **轻量审查**：三维度都查、单轮、不循环；跳过"修复方案可执行性"项；根不可达只查证据链与假设自洽，根可达加核行号；偏差回写一次、不再复审，止于此交付、不进 5/6/7。

**审查者**：若派子 agent 审查，须是全新 agent、不复用步骤 2/3 的分析推理（对抗式复核）；也可主对话串行自查，但逐条核验、不抽样。

- **完整性维度**（有没有漏）
  - 步骤 2 识别到的每个问题是否都落了独立文档。
  - 根因是否附齐日志关键证据行；源码定位（文件 + 实际行号）是否齐全。
  - 跨模块关联是否标全（上下游模块 + 调用路径）。

- **合理性维度**（方向对不对）
  - 排查方向 / 修复方案是否贴合代码真实意图，而非机械对号入座。
  - 是否有更简结论；是否过度归因（把症状当根因、把单一现象推广成系统性结论）。

- **正确性维度**（站不站得住）
  - 根因与所附日志证据是否一致、能否由证据推出。
  - 源码行号是否核验到**实际行**（报告行常偏移，以实际行为准），不一致处标注「报告 X / 实际 Y」。
  - 修复方案是否可执行、有无自相矛盾。

**审查是循环，不是单遍：** 一轮三维度审查 → 合并偏差 → 回写文档 → **再跑一轮三维度审查**。如此往复，直到**连续一轮零新问题**才停止（「审干」）。以「最新文档」为审查对象，不能只看上轮没动的条目——改一处可能连带暴露别处的偏差。给用户的最终汇报里注明共审了几轮。

> **安全上限**：审查最多 **5 轮**。若 5 轮仍有审出新问题，停下、把未决项列出**交用户裁决**（可能是结论本身需重新论证），不无限重审。

### 步骤 5：实施修改（必须经用户同意）

步骤 3/4 只产出并审干「修复方案文档」，**不碰真实源码**。到这一步：若 `范围=全流程`（已预授权全部实施），跳过下方弹问、按全部实施直接进入 Edit；否则先停下，**征得用户明确同意后才动代码**：

- 用 AskUserQuestion 询问：是否现在按修复方案文档实施代码修改？选项：全部实施 / 选择性实施（按问题或文件筛选）/ 暂不实施（仅保留方案）。（全流程跳过此步）
- 同意后，按方案对真实源码文件用 Edit 逐条改：
  - 先 Read 目标文件再 Edit（Edit 要求文件已读；顺便再次核验当前行号，源码可能在分析期间有变动）。
  - 头文件声明 + .cpp 实现配套的改动一起完成，不留半成品。
  - 真实工程改动不可逆、影响大：逐条确认、只改方案内列明的项、**不顺手改周边无关代码**、不引入风格不一致。
  - 保留一份修改清单（文件 + 行 + 改动摘要），供步骤 6 审查、步骤 7 commit message 与最终汇报。
- 用户未同意 → 到此结束，交付修复方案文档即可，不动代码。

### 步骤 6：改后多遍审查（对实际 diff，循环直到审干）

代码改完后，对**实际 diff**跑三维度循环审查——审查对象是「真实落地后的代码」(diff)，**修复方案文档作对照基准**(判每条改动是否落实、有无跑偏；审查者须读这两份)。修一处可能引出新问题，故循环到审干。维度检查项见下方。

- **完整性维度**：修复方案文档里列的每条改动是否都落实了；头文件 / 配套 free/delete 等连带项有无漏改；同文件同类问题改完后口径是否一致（如 NULL 全改 nullptr，不能留半边）。
- **合理性维度**：实际改动是否贴合既有代码风格（命名、注释密度、错误处理惯用法）；是否有比方案更简的实现已在改动中体现。
- **正确性维度**：默认**静态走查（读代码 + 推理，不编译）**，检查：
  - 语法/头文件依赖/类型、new/delete 配对、`delete` vs `delete[]`、RAII 析构顺序、`static_cast` 保留 const 是否正确
  - 实际 diff 有无引入新 UB/泄漏/双重释放/回归
  - 车载构建（soong `m`）耗时长，**不自动触发编译**；仅当用户明确"构建环境已就绪且要编译验证"时才跑，否则交付时标注"未编译验证、需人工/CI 复核"

发现偏差 → 交主对话回写 diff → 重新审查（审查 agent 只审不改，回写由主对话做；每轮以最新 diff 为对象）。循环到「连续一轮零新问题」才停（与步骤 4 同判据）。把最终修改清单与审查轮数纳入给用户的汇报。

> **安全上限**：步骤 6 同样最多 **5 轮**（与步骤 4 各自独立计数）。若 5 轮内某条改动始终正确性不达标（启用编译验证时含编译不过）——先尝试反向修正该条；仍不收敛则把未决项列出交用户裁决，**不留半坏代码静默收场**。

### 步骤 7：生成 commit message

步骤 6 审干后，把本次实际落地的改动按 `reference/commit_guide.md` 的要求整理成源码仓库的 commit message 并提交——该文件标注当前是否可自动提交及环境约束；支持则执行提交，不支持则只在对话内生成、由用户手动提交。提交不可逆且对外，支持自动提交时须经用户明确授权后执行。

**触发条件**：仅当步骤 5 已实际实施代码改动（用户同意或 `范围=全流程` 预授权，且代码已改）且步骤 6 **审干**（连续一轮零新问题）时才进入本步。步骤 5 用户选「暂不实施」→ 未改代码，不进入；步骤 6 撞安全上限未收敛 → 不进入，代码留工作区不提交，随未决项一并交用户裁决。`analysis` 类型不进 5/6，自然不进本步。

## 活体调试（按需）

活体调试用 adb 补充日志覆盖不到的现场信息（dumpsys 运行时状态、实时 logcat 等）。活体调试是否可用在分析开始前一次性问清、不在分析过程中主动问（其他不清楚之处仍按步骤 2 主动询问）；用户仍可在后续任一轮次边界主动告知可用。

- `可用` = 用户确认当前连接的是目标设备（SOC 平台无法从 adb 属性可靠推断）；agent **仅在日志不足时按需**跑只读诊断，不代表一定跑。
- `不可用` = 不跑活体诊断。

**诊断命令、证据处置与操作边界**：见 `reference/adb_diagnostics.md`（dumpsys/logcat 清单、子 agent 跑大输出、当前快照 vs 故障时刻的旁证定位、adb 失败处理、权限分层）。**写操作（改运行参数 / 推库）需用户授权或协助（root / remount），不外包给子 agent。**

## 日志标签 → 模块

通用标签（AudioFlinger/AudioPolicy/CarAudio/AAudio）直接判，模块即 paths.md 同名章节，不另列表。

厂商标签的 module_id = 侧前缀 + 模块名：`gvm_`=Android 侧、`pvm_`=PVM（Linux/QNX）侧、`adsp_`=Foundation/ADSP 固件层（仅 MT8091；SA8397 虽有 ADSP/Gunyah 但 paths.md 未索引 foundation 层）。各 paths.md 章节标题以 `· <module_id>` 标注、章节内登记该模块日志标签——步骤 2 全读 paths.md 即拿到映射，见标签直接落地对应章节；同名异义（AHAL 各形态）的区分见各章节标签行。

识别规则：`TAG:` 前缀取冒号前部分；进程名/字面前缀（非 `TAG:`）；dmesg 字面前缀（`[IPI]`/`[ADSP]` 等）。

不登记标签的底层模块（无单一前缀，直接读 paths.md 章节）：`pvm_alsa_lib`（MT8091，prebuilt 无源码）、`pvm_alsa_kernel`（MT8091/SA8397，经源码确认无单一前缀）、`adsp_firmware`/`adsp_hypervisor`（MT8091 Foundation；hypervisor 无源码）、`gvm_pal_awe_plugin`（SA8397）等 PAL 子模块、`pvm_hab_awe_backend`（SA8397，上层 prebuilt daemon、靠 `journalctl` 推断）。未登记不等于找不到路径。
