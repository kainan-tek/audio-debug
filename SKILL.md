# 车载音频日志分析与调试 Skill

`/audio-debug` — 分析 AAOS 音频系统日志，定位故障根因并给出排查建议；需要时可用 adb 活体调试补充现场信息（见「活体调试」章节）。

**日志输入**：直接读取用户给出的日志文件（扩展名不限——`.log`/`.txt`/`.qlog`/无扩展名等均可，按内容判断是否为日志）。若提供的是日志全量包（`.zip`/`.7z`），请用户先解压出日志文件再提供。

> **路径约定**：本 Skill 内所有相对路径（`platforms/`、`reference/`、`issues/`、`output/`、各 `paths.md`）一律相对本 Skill 自身目录 `${CLAUDE_SKILL_DIR}` 解析，与会话 cwd 无关——任意目录启动 `claude` 均可正常工作。

> **适用范围**：本 Skill 主线是日志驱动的故障定位（见下方「分析工作流」）。其资产在功能开发时也可直接查用，不必走 log→根因 的循环——`platforms/*_paths.md` 全栈源码路径图定位改动落点，`reference/` 架构与调试参考建立全栈心智模型，`output/reports/` 避开同区域历史坑。开发场景按需查阅上述资产即可。

## 分析工作流

调用 `/audio-debug` 时，需获取以下三项（缺一不可）：

- **SOC 平台**（SA8295、MT8091 或 SA8397；SA8797 按 SA8397 流程处理）（必须，不应从日志推断）
- **日志文件路径**（必须）
- **问题描述**（必须，越详细越好——现象、发生时间、触发条件、影响范围）

两种方式提供三项（二选一，独立、不混用不合并）：

- **命令行**：`/audio-debug <平台> <日志路径> <问题描述>`

  ```
  /audio-debug SA8295 /data/logs/audio.log "AudioFlinger underrun 问题"    ← 三项齐全，直接开始
  /audio-debug SA8295 /data/logs/audio.log                                 ← 缺问题描述，追问
  /audio-debug SA8295                                                      ← 缺日志路径和问题描述，追问
  /audio-debug                                                             ← 三项都缺，逐项追问
  ```

- **文件**：三项预填在 `${CLAUDE_SKILL_DIR}/issues/issue_info.txt`。调用时在提示词中说明「从文件读取」即可，agent 读该文件取三项。格式（标签用全角冒号，`问题描述` 后为自由文本可多行）：

  ```
  以下是问题所需信息的描述：
  SOC平台：<SA8295 | MT8091 | SA8397>
  日志文件：<日志文件路径，扩展名不限>
  问题描述：<现象、发生时间、触发条件、影响范围，越详细越好，可多行>
  补充信息：<可选，专业背景/已知约束/怀疑方向等，agent 作上下文参考、非结论，与日志冲突以日志为准>  # 选填，可留空，纯背景、不弹问
  活体调试：<可用 | 不可用>  # 选填，不填则分析开始前弹问
  范围：<暂不实施代码修改（仅步骤1-3） | 全流程>  # 选填，不填则到步骤4弹问
  ```

三项齐全 → 进入分析流程（开始前先问清活体调试是否可用，见下）。缺少任何一项 → 仅追问缺失项，不重复已提供的信息（两种方式均适用）。

分析过程中如有不清楚之处或需要补充资料（日志片段、现象细节、复现步骤等），Agent 应主动向用户询问，不应跳过或猜测。

**分析开始前一次性问清活体调试是否可用**：用 AskUserQuestion 问——`可用（已确认连接目标设备）` / `不可用（不跑活体诊断）`。规则见「活体调试」章节。

### 步骤 1：问题分析

用户应给出具体的日志文件（一个或多个，扩展名不限）；不确定该看哪个就问，不猜不遍历。**多文件按时间戳对齐**——跨文件追同一事件传播链，不孤立看单文件。

**先核对平台一致性**：以行首反复出现的前缀为准（`TAG:` 或进程名/字面前缀，如 `audio_service`/`[IPI]`；不看消息正文零散子串，如路径/变量名里碰巧的 `pal:`）。若日志**主体标签**与所选 SOC 平台矛盾——如 SA8295 却是 MTK 的 `MTKAdapterLayer`/`AHAL_Module`（无 `_QTI`）、MT8091 却是 QCOM 的 `PAL:`/`AGM:`/`AHAL_*_QTI`——先向用户确认平台。平台错则标签表与源码路径整套错。

**分析范围** = 报告现象的时间窗，不假装扫整条 boot。时间窗已给→限定到该窗；未给→追问，不自行猜。窗口可全读则读透，过大才退到"搜索定位 + 定点精读"。

读日志、定位根因——**方法由 agent 自主，工具只提供导航**：

- **先全读对应平台 paths.md**：拿音频全栈源码路径索引 + 各章节 module_id，供跨模块追溯与源码定位；可达/不可达两种模式都需要（不可达据此推断可疑路径）。
- paths.md 按 GVM / PVM /（mt8091）Foundation 分段，**各侧源码根路径不同**（见文件头，按章节侧标志拼根）。
- **同名模块可能在两侧各一份**（如 mt8091 的 ALSA Kernel/ADSP/Audio IPI 驱动），勿混；GVM 侧同名副本未标 module_id，须读章节标题。
- **探测源码可达性定模式**：试 Read 一个 paths.md 登记的源码文件，能读→**可达模式**（一次定下整次沿用，中途不因单文件漂移翻模式）。
- **漂移 fallback**：不能读先别判不可达——paths.md 是静态索引、源码在持续更新、路径可能漂移，先在源码树用 Glob 按 paths.md 章节登记的源码文件名定位实际位置再试，找得到仍算可达（paths.md 过时），都找不到→**不可达模式**。两种都常见，不预设常态。
- **paths.md 每模块只列核心文件，非穷举**：日志引用的源码文件若未登记，别判「不存在/不可达」——以同模块已登记文件的路径为目录锚点 Glob 该模块源码目录定位。
- **不可达是正式模式而非兜底**：用"证据链 + 基于标签表与 paths.md 推断的可疑路径"定位，精度到路径不到行。
- **知识图（暂不支持）**：当前停用，源码定位走 paths.md 章节路径 + 直接 Read 源码。
- **日志标签 → 模块**：通用标签（AudioFlinger/AudioPolicy/CarAudio/AAudio）直接判，厂商特有标签查下方「日志标签→模块映射」表。

**完成判据（方法自主，"做没做完"按此判）**：

- **不停在首个表面命中**：分析范围内逐一识别所有独立问题，不因找到第一个就收工。
- **五件套缺一不算完成**：根因（可为待证假设）+ 源码文件 + 源码定位 + 支撑证据（日志行）+ 跨模块关联（传播链/调用路径）。源码定位按模式给——可达给文件路径 + 实际行号，不可达给 paths.md 路径 +「源码未验证」，不硬凑。跨模块关联须实际追过，查无证据才能标"无"，不得省略。
- **范围诚实**：能力外的（日志超大未穷举、源码不可达）或未发现匹配故障时，显式标注提醒人工确认，不静默交差、不硬凑根因。

输出：每条独立问题的五件套——供步骤 2 写入文档。

### 步骤 2：综合结论 → 写修复方案文档

**查无问题**：若步骤 1 未识别到任何独立问题（分析范围内无匹配故障现象），不写文档、不硬凑根因；在对话内说明查无并给判断——预期行为/误报则如实说明，证据不足则建议补日志/确认时间窗/提供复现步骤——交用户判断，到此结束。

综合所有模块分析结果，**每个识别到的独立问题（一个问题一个文件）**写入 `${CLAUDE_SKILL_DIR}/output/reports/<主题>_<类型>.md`。主题用英文 snake_case（如 `adsp_timeout`）。类型按根因确定度：根因确定 → `fix_plan`（修复方案，需源码可达）；根因不确定 → `analysis`（标「根因待证」）；源码不可达一律 `analysis`（标「源码未验证」）。如 `adsp_timeout_analysis.md`、`audioflinger_sched_rr_fix_plan.md`。**不按 bug ID 命名**；现有 output/reports/ 历史文件命名风格不一，新建按本规则即可，不回改历史文件。

每个文档必含字段：

- 故障类型
- 根因（附日志关键证据行）
- 源码定位：按步骤 1 五件套规则——可达给文件路径 + 实际行号；不可达给 paths.md 路径 +「源码未验证」
- 排查方向 / 修复方案
- 跨模块关联（上下游模块 + 调用路径），有则标，无则省

对话内仅给**摘要**（处理了几个问题、各自文档路径），不重复文档全文。

**闸门**：每个文档独立分流——`fix_plan`（根因确定且源码可达）→ 步骤 3 **全量审查** → 步骤 4/5/6；`analysis`（根因待证或源码不可达）→ 步骤 3 **轻量审查** → 止于此交付（不进 4/5/6），相应标「根因待证」/「源码未验证」。升 `fix_plan` 由用户补足所缺（证据 / 源码可达性）后重新发起 /audio-debug。

### 步骤 3：多遍审查（`fix_plan` 全量 / `analysis` 轻量）

审查与「分析」是两回事：分析是给结论，审查是**用独立视角挑结论的毛病**。对步骤 2 写出的每个文档，按类型走不同审查强度：`fix_plan` → **全量审查**（下方三维度逐条核验，循环到审干）；`analysis` → **轻量审查**（单轮、仅正确性维度、不循环；不可达文档无源码可核，只查日志证据链与假设逻辑是否自洽；审出偏差由主对话回写一次、不再复审，止于此交付，不进步骤 4/5）。**若派子 agent 审查，必须是全新 agent、不复用步骤 1/2 的分析推理**（避免既当运动员又当裁判、确认偏误；对抗式复核）。也可主对话串行自查，但**逐条核验、不抽样**。

- **完整性维度**（有没有漏）
  - 步骤 1 识别到的每个问题是否都落了独立文档。
  - 根因是否附齐日志关键证据行；源码定位（文件 + 实际行号）是否齐全。
  - 跨模块关联是否标全（上下游模块 + 调用路径）。

- **合理性维度**（方向对不对）
  - 排查方向 / 修复方案是否贴合代码真实意图，而非机械对号入座。
  - 是否有更简结论；是否过度归因（把症状当根因、把单一现象推广成系统性结论）。

- **正确性维度**（站不站得住）
  - 根因与所附日志证据是否一致、能否由证据推出。
  - 源码行号是否核验到**实际行**（报告行常偏移，以实际行为准），不一致处标注「报告 X / 实际 Y」。
  - 修复方案是否可执行、有无自相矛盾。

**审查是循环，不是单遍：** 一轮三维度审查 → 合并偏差 → 回写文档 → **再跑一轮三维度审查**。如此往复，直到**连续一轮零新问题**才停止（「审干」）。以「最新文档」为审查对象，不能只看上轮没动的条目——改一处可能连带暴露别处的偏差。**三个维度全过、且再也审不出新问题，才算一条定稿。** 给用户的最终汇报里注明共审了几轮。

> **安全上限**：审查最多 **5 轮**。若 5 轮仍有审出新问题，停下、把未决项列出**交用户裁决**（可能是结论本身需重新论证），不无限重审。

### 步骤 4：实施修改（必须经用户同意）

步骤 2/3 只产出并审干「修复方案文档」，**不碰真实源码**。到这一步先停下，**征得用户明确同意后才动代码**：

- 用 AskUserQuestion 询问：是否现在按修复方案文档实施代码修改？选项：全部实施 / 选择性实施（按问题或文件筛选）/ 暂不实施（仅保留方案）。
- 同意后，按方案对真实源码文件用 Edit 逐条改：
  - 先 Read 目标文件再 Edit（Edit 要求文件已读；顺便再次核验当前行号，源码可能在分析期间有变动）。
  - 头文件声明 + .cpp 实现配套的改动一起完成，不留半成品。
  - 真实工程改动不可逆、影响大：逐条确认、只改方案内列明的项、**不顺手改周边无关代码**、不引入风格不一致。
  - 保留一份修改清单（文件 + 行 + 改动摘要），供步骤 5 审查、步骤 6 commit message 与最终汇报。
- 用户未同意 → 到此结束，交付修复方案文档即可，不动代码。

### 步骤 5：改后多遍审查（对实际 diff，循环直到审干）

代码改完后，对**实际 diff**跑三维度循环审查——这次审的不是「方案文档」，而是「真实落地后的代码」。修一处可能引出新问题，故循环到审干（全新 agent 与循环判据同步骤 3）。维度检查项见下方（与步骤 3 的清单不同，针对 diff）。

- **完整性维度**：修复方案文档里列的每条改动是否都落实了；头文件 / 配套 free/delete 等连带项有无漏改；同文件同类问题改完后口径是否一致（如 NULL 全改 nullptr，不能留半边）。
- **合理性维度**：实际改动是否贴合既有代码风格（命名、注释密度、错误处理惯用法）；是否有比方案更简的实现已在改动中体现。
- **正确性维度**：默认**静态走查（读代码 + 推理，不编译）**，检查：
  - 语法/头文件依赖/类型、new/delete 配对、`delete` vs `delete[]`、RAII 析构顺序、`static_cast` 保留 const 是否正确
  - 实际 diff 有无引入新 UB/泄漏/双重释放/回归
  - 车载构建（soong `m`）耗时长，**不自动触发编译**；仅当用户明确"构建环境已就绪且要编译验证"时才跑，否则交付时标注"未编译验证、需人工/CI 复核"

发现偏差 → 交主对话回写 diff → 重新审查（审查 agent 只审不改，回写由主对话做；每轮以最新 diff 为对象）。循环到「连续一轮零新问题」才停（与步骤 3 同判据）。把最终修改清单与审查轮数纳入给用户的汇报。

> **安全上限**：步骤 5 同样最多 **5 轮**（与步骤 3 各自独立计数）。若 5 轮内某条改动始终正确性不达标（启用编译验证时含编译不过）——先尝试反向修正该条；仍不收敛则把未决项列出交用户裁决，**不留半坏代码静默收场**。

### 步骤 6：生成 commit message（只生成不提交）

步骤 5 审干后，把本次实际落地的改动整理成源码仓库的 commit message，**仅在对话内输出，不跑任何 git 命令**——由用户手动提交到真实源码仓库（远端映射盘无法直接 `git commit`）。

**简洁**：commit message 以说清"问题 + 解法"为度，不堆砌日志原文、不铺陈分析过程。一句话概括精炼到一行；Root cause / Solution 只保留讲清场景、逻辑、现象所必需的信息。

**触发条件**：仅当步骤 4 已实际实施代码改动（用户同意且代码已改）且步骤 5 **审干**（连续一轮零新问题）时才生成。步骤 4 用户选「暂不实施」→ 未改代码，不生成；步骤 5 撞安全上限未收敛 → 不生成，随未决项一并交用户裁决。`analysis` 类型不进 4/5，自然不进本步。

**整次一条**：一次 /audio-debug 改了多个问题/文件，合并成一条 commit message。多个独立问题时，`Root cause` / `Solution` 按问题分条枚举。

按源码仓库模板填，字段映射：

- `FIX XXX [XXX][XXX][XXX] <一句话概括>`：XXX（票号 / 模块标签）由用户填，skill 不动；一句话概括由 skill 起草——概括本次改了什么、解决了什么现象，**不拷贝 bug 票标题**。素材取自最终修改清单（步骤 5 审查回写后）+ 步骤 2 根因。
- `Project: XXX`：用户填，skill 不动。
- `Root cause:`：bug 分支——清晰描述 在什么场景下 + 什么代码逻辑 + 导致什么问题现象。素材取自步骤 1 根因 / 故障类型 / 日志关键证据行。
- `Solution:`：用什么代码逻辑解决了问题。素材取自最终修改清单（步骤 5 审查回写后，含文件 + 改动摘要）+ 步骤 2 修复方案。
- `Test suggest:`：自测报告链接 skill 无从知晓，留占位 `<待填：自测报告链接>`，由用户补。

**只写最终落地项**：已回退的改动不写入 Solution，commit message 只反映步骤 5 后最终保留的代码改动。

输出：完整 commit message 在对话内给出，用户复制后去源码仓库手动 `git commit`。

## 活体调试（按需）

活体调试用 adb 补充日志覆盖不到的现场信息（dumpsys 运行时状态、实时 logcat 等）。**是否可用在分析开始前一次性问清**（见「分析工作流」），不在分析过程中主动问；用户仍可在后续任一轮次边界主动告知可用。

- `可用` = 用户确认当前连接的是目标设备（SOC 平台无法从 adb 属性可靠推断）；agent **仅在日志不足时按需**跑只读诊断，不代表一定跑。
- `不可用` = 不跑活体诊断。
- 跑诊断时若 adb 不可用 / 设备掉线 / 命令失败：告知用户一次并停用活体诊断、继续分析，不重试、不再次追问。
- 活体诊断取的是**当前**设备状态快照，与日志记录的**故障时刻**未必一致（设备可能已重启、问题未必在复现）——只作旁证交叉印证，不当故障时刻的直接证据，除非问题正在现场复现。

只读诊断（普通 adb shell，无需 root；可派子 agent 跑，避免大输出灌爆主上下文，子 agent 须保留关键原始行作证据）：

- `adb shell dumpsys media.audio_flinger` — AudioFlinger 运行时状态（Track/线程/缓冲区健康度）
- `adb shell dumpsys media.audio_policy` — AudioPolicy 策略与路由（可用设备/路由决策/AudioPatch）
- `adb shell dumpsys car_audio` — CarAudio 车载音频配置（音区/设备/音量组）
- `adb shell logcat -b all -d -s AudioFlinger AudioPolicy CarAudio` — 近期音频日志
- 其他无副作用的诊断命令（按需选用，不必全跑）

**操作边界**（按权限分层）：

- 只读诊断（dumpsys / logcat 等）：自主执行（或派子 agent），无需 root
- 重启服务（killall audioserver 等）/ 修改运行参数（setprop、settings put 等）：需用户授权 + **root**，仍在主对话走授权，不外包给子 agent
- 推库 / 改系统文件：需用户协助 + **remount**
